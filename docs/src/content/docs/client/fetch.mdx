---
title: Fetch Client
description: Create a fully-typed HTTP client from an api.
---

The fetch client generates typed methods for every endpoint in an api. No code generation or compile step — just TypeScript inference. It owns the whole request lifecycle: URL building, JSON serialization, the `fetch` call, response parsing, and error mapping.

## Basic Usage

```ts
import { createFetchClient } from "@jayalfredprufrock/handshake/client";
import { api } from "./contract";

const client = createFetchClient(api, { baseUrl: "http://localhost:3000" });

const users = await client.listUsers();
const user = await client.getUser({ id: "1" });
const created = await client.createUser({ name: "Alice", email: "alice@example.com" });
```

`fetch` defaults to `globalThis.fetch` — there's nothing to wire up.

Endpoints marked `internal: true` are omitted from the generated client, so they never appear as callable methods.

## `createFetchClient(api, config)`

### Config

```ts
interface FetchClientConfig {
  baseUrl: string;
  fetch?: typeof globalThis.fetch; // defaults to globalThis.fetch
  handleRequest?: (ctx: RequestContext) => Request | void | Promise<Request | void>;
  handleResponse?: (ctx: ResponseContext) => void | Promise<void>;
  retry?: (ctx: ResponseContext) => boolean | Promise<boolean>;
}
```

#### `baseUrl`

Combined with the api's base path, the group's base path, and the endpoint path to form the full URL.

```ts
// api basePath "/api", group basePath "/users", endpoint path "/:id", baseUrl "http://localhost:3000"
// → GET http://localhost:3000/api/users/1
```

#### `fetch`

The fetch implementation (default `globalThis.fetch`). Use it to supply a different runtime's fetch, a mock in tests, or a fetch configured with a proxy/agent. It is a **standard** `fetch` — the client handles serialization and parsing around it.

#### `handleRequest`

Inspect, mutate, or replace the outgoing `Request` before it's sent — typically to add auth headers. It re-runs on every attempt, so a refreshed token is picked up on retry.

```ts
handleRequest: (ctx) => ctx.request.headers.set("authorization", `Bearer ${tokens.access}`),
```

#### `handleResponse`

Runs on every response. Throw to **reject** an otherwise-successful response (e.g. an AWS WAF `202` challenge) or to reshape an error; return to accept the default disposition.

```ts
handleResponse: (ctx) => {
  if (ctx.response?.status === 202) throw new WafChallenge();
},
```

#### `retry`

Decide whether to retry a failed attempt — return `true` to re-issue the request. `ctx.error` is the `ApiError` (for a non-OK response), the error your `handleResponse` threw, or a raw network error.

```ts
retry: async (ctx) => {
  if (ctx.error instanceof WafChallenge) {
    await solveChallenge();
    return true;
  }
  if (api.isError(ctx.error, "TOKEN_EXPIRED")) {
    await tokens.refresh();
    return true;
  }
  return false;
},
```

## Method Signatures

The generated method signature adapts to the endpoint:

```ts
client.updateUser(params, body, options); // params + body
client.getUser(params, options); // params only
client.createUser(body, options); // body only
client.listUsers(options); // neither
```

The `options` argument — and each of its fields (`query`, `headers`) — is **required only when the corresponding schema has a required property**, and optional otherwise.

## Client Options

```ts
interface ClientOptions<E extends Endpoint> {
  query?: InferSchema<E["query"]>;
  headers?: InferSchema<E["headers"]>;
  request?: Omit<RequestInit, "method" | "body">;
}
```

### `query`

Query parameters, typed from the endpoint's `query` schema. Array values serialize as repeated keys.

```ts
await client.search({ query: { q: "hello", page: 2, tags: ["ts", "api"] } });
// → GET /search?q=hello&page=2&tags=ts&tags=api
```

### `headers`

Request headers, typed from the endpoint's `headers` schema (declare header names lowercase). Merged into the request; `request.headers` and `handleRequest` override them.

```ts
await client.secret({ headers: { "x-api-key": key } });
```

### `request`

Additional `RequestInit` fields (abort signal, credentials, escape-hatch headers).

```ts
const controller = new AbortController();
await client.listUsers({ request: { signal: controller.signal } });
```

## Errors

A handshake error response is thrown as an `ApiError`, which mirrors the wire envelope — `{ code, status, message, details }` — plus a client-only `response`. The server brands its envelope with `kind: "HANDSHAKE"`, so the client reconstructs the `ApiError` 1:1. Any **other** non-OK response (a proxy, gateway, or non-handshake backend) is thrown as an `HttpError` (`{ status, body, response }`) — never an `ApiError`. Network failures propagate as-is. Recognize and narrow api errors with `api.isError`:

```ts
import { api } from "./contract";
import { HttpError } from "@jayalfredprufrock/handshake/client";

try {
  await client.getUser({ id: "42" });
} catch (err) {
  if (api.isError(err)) {
    switch (err.code) {
      case "NOT_FOUND":
        return null;
    }
  } else if (err instanceof HttpError) {
    throw err; // non-handshake response (status/body available)
  } else {
    throw err; // network / unexpected
  }

  // or check a specific code
  if (api.isError(err, "CONFLICT")) err.details.conflictingId; // typed
}
```

The api is also reachable from the client as `client.$api`, so `client.$api.isError(...)` works where you only hold the client.

## Endpoint Metadata

Each method also exposes the original endpoint definition:

```ts
client.getUser.method; // "GET"
client.getUser.path; // "/users/:id"
client.getUser.meta; // { auth: "required" } (if defined)
```
