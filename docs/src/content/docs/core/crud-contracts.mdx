---
title: CRUD Contracts
description: Generate standard CRUD endpoints from a single TypeBox schema.
---

For common REST resources, `createCrud` generates a full set of CRUD endpoints (`get`, `list`, `create`, `update`, `delete`) from a single TypeBox schema and a configuration object.

## Basic Usage

```ts
import { createContract, createCrud } from "@jayalfredprufrock/handshake/contract";
import { Type } from "typebox";

const User = Type.Object({
  id: Type.String(),
  name: Type.String(),
  email: Type.String(),
  role: Type.String(),
  passwordHash: Type.String(),
  createdAt: Type.String(),
});

const userEndpoints = createCrud(User, {
  params: ["id"],
});

const contract = createContract("/users", userEndpoints);
```

This generates five endpoints, all relative to the wrapping contract's base path:

| Name     | Method   | Path   | Body                          | Response        |
| -------- | -------- | ------ | ----------------------------- | --------------- |
| `get`    | `GET`    | `/:id` | —                             | Full schema     |
| `list`   | `GET`    | `/`    | —                             | Array of schema |
| `create` | `POST`   | `/`    | Full schema (minus params)    | Full schema     |
| `update` | `PATCH`  | `/:id` | Partial schema (minus params) | Full schema     |
| `delete` | `DELETE` | `/:id` | —                             | Full schema     |

With `basePath: "/users"` on the wrapping contract, final URLs resolve to `/users`, `/users/:id`, etc.

## Configuration

The second argument to `createCrud` is a configuration object:

### `params`

**Required.** An array of property keys from the schema that identify an individual resource. These become path parameters for `get`, `update`, and `delete`, and are excluded from the `create` body.

```ts
params: ["id"];
// Path: /:id

params: ["tenantId", "id"];
// Path: /:tenantId/:id
```

### `hidden`

**Optional.** Properties that should be excluded from all responses and from the `create`/`update` bodies. Useful for sensitive fields that should never leave the server.

```ts
hidden: ["passwordHash"];
```

### `readonly`

**Optional.** Properties that appear in responses but cannot be set during `create` or `update`. Useful for server-generated fields.

```ts
readonly: ["createdAt"];
```

### `immutable`

**Optional.** Properties that can be set during `create` but cannot be changed during `update`. They still appear in responses.

```ts
immutable: ["role"];
```

## Full Example

```ts
const userEndpoints = createCrud(User, {
  params: ["id"],
  hidden: ["passwordHash"],
  readonly: ["id", "createdAt"],
  immutable: ["email"],
});
```

This produces:

| Endpoint | Body fields                   | Response fields                            |
| -------- | ----------------------------- | ------------------------------------------ |
| `create` | `name`, `email`, `role`       | `id`, `name`, `email`, `role`, `createdAt` |
| `update` | `name`, `role` (all optional) | `id`, `name`, `email`, `role`, `createdAt` |
| `get`    | —                             | `id`, `name`, `email`, `role`, `createdAt` |
| `list`   | —                             | Array of the above                         |
| `delete` | —                             | `id`, `name`, `email`, `role`, `createdAt` |

- `passwordHash` is excluded everywhere (`hidden`)
- `id` and `createdAt` appear in responses but not in `create` or `update` bodies (`readonly`)
- `email` appears in `create` but not in `update` (`immutable`)
- `update` body fields are all optional (partial update semantics)

## Using with `createContract`

The result of `createCrud` is a plain object of endpoint definitions — each endpoint's `path` is relative, so wrap the result in a `createContract` call with a base path:

```ts
import { createContract, createCrud } from "@jayalfredprufrock/handshake/contract";

const usersContract = createContract("/users", createCrud(User, { params: ["id"] }));
const postsContract = createContract("/posts", createCrud(Post, { params: ["id"] }));
```

Each resource produces its own contract with identical endpoint keys (`get`, `list`, …), so treat them as separate contracts rather than spreading them into a single one. Use `implementContract` and pass the resulting route modules to `createHonoApp` to serve them from the same app — see [Splitting contracts across files](/handshake/server/hono/).
