A contract defines one resource â€” every endpoint (method, path, params, query, headers, body, and response) plus the errors those endpoints can return. Contracts are building blocks: you compose one or more of them into an **api** with [`createApi`](#composing-contracts-into-an-api), and that api is the single source of truth the server adapter, client, and OpenAPI generator are all derived from.

## Creating a Contract

Call it as `createContract(endpoints)`, `createContract(basePath, endpoints)`, or `createContract(basePath, endpoints, options)`. The base path prefixes all endpoint paths (defaults to `"/"`); passing `options` requires an explicit base path.

```ts
import { createContract } from "@jayalfredprufrock/handshake/contract";
import { Type } from "typebox";

const contract = createContract("/api", {
  getUser: {
    method: "GET",
    path: "/users/:id",
    params: Type.Object({ id: Type.String() }),
    response: Type.Object({ id: Type.String(), name: Type.String() }),
  },
});
```

## Endpoint Definition

### `method`

**Required.** `"GET"`, `"POST"`, `"PATCH"`, or `"DELETE"`.

### `path`

**Required.** Path relative to the base path. Path parameters use `:param` syntax.

### `response`

**Required.** A TypeBox schema for the response body â€” used for both type inference and runtime response validation.

### `responseCode`

**Optional.** The HTTP status for a successful response (defaults to `200`).

```ts
createUser: { method: "POST", path: "/users", body, response, responseCode: 201 }
```

### `params`

**Optional.** Schema for path parameters (a `Type.Object` keyed by `:param` segments). String values from the URL are coerced to match (e.g. `"42"` â†’ `42`).

### `query`

**Optional.** Schema for query-string parameters. Coerced like params; array schemas support repeated keys (`?tags=a&tags=b`).

### `headers`

**Optional.** Schema for request headers. **Header names must be declared in lowercase.** Coerced like params; headers not named in the schema are ignored.

```ts
headers: Type.Object({ "x-api-key": Type.String() });
```

### `body`

**Optional.** Schema for the JSON request body. Validated strictly on the server â€” missing or extra properties produce a `VALIDATION_ERROR` (400).

### `description` / `meta`

**Optional.** A human-readable description and an arbitrary metadata object for documentation or custom tooling.

By default `meta` accepts any object. To type it, augment the exported `EndpointMeta` interface via declaration merging â€” target the **`/contract`** subpath (where the interface ships):

```ts
declare module "@jayalfredprufrock/handshake/contract" {
  interface EndpointMeta {
    auth: boolean; // required
    scope?: string; // optional
  }
}
```

With that in place:

- **Declared keys are typed** everywhere `meta` is read â€” `endpoint.meta.auth` is `boolean`, `meta.scope` is `string | undefined`.
- **A required key makes `meta` required on every endpoint.** Declaring `auth: boolean` (no `?`) means an endpoint with no `meta`, or a `meta` missing `auth`, is a compile error. Declare keys as optional (`auth?`) to keep `meta` itself optional.
- **Extra keys are still allowed.** `EndpointMeta` extends `Record<string, any>`, so augmentation is additive â€” you can still attach ad-hoc keys beyond the ones you declared.
- **Contract- and api-level default `meta`** (below) are typed against this same interface.

### `internal`

**Optional.** Marks the endpoint as **server-only**. The server adapters still implement and serve it, but it is excluded from the generated fetch client (both its type and the runtime object) and from the OpenAPI document â€” useful for endpoints that should only be called from other trusted services, not exposed to client consumers.

```ts
purgeCache: {
  method: "DELETE",
  path: "/cache",
  response: Type.Null(),
  internal: true,
}
```

## Errors

Errors are declared once on the contract as a **code-keyed map**. Each `code` maps to its HTTP `status` and an optional `details` schema. Errors are **contract-wide** (any endpoint may return any of them):

```ts
const contract = createContract(
  "/api",
  { transfer: { method: "POST", path: "/transfer", body, response } },
  {
    errors: {
      UNAUTHORIZED: { status: 401 },
      NOT_FOUND: { status: 404 },
      CONFLICT: { status: 409, details: Type.Object({ conflictingId: Type.String() }) },
    },
  },
);
```

Codes are unique by construction (map keys). The codes `VALIDATION_ERROR` (400) and `UNKNOWN_ERROR` (500) are **reserved** â€” they're emitted by the framework and **cannot be declared** in your map (declaring one throws), but they're always available to `contract.error("VALIDATION_ERROR", message, issues?)` (where the optional `issues` is a `{ path?, keyword?, message }[]`) / `contract.error("UNKNOWN_ERROR", message)` and recognized by `contract.isError`, since the client can receive them on any endpoint.

The contract exposes two typed helpers:

- **`contract.error("CODE", message, details?)`** â€” builds an `ApiError`; `throw` the result. The `status` comes from the code's definition, the second argument is the `Error.message`, and the third is the code's `details` payload (required only when the code declares a `details` schema). Undeclared codes and bad details are compile errors.
- **`contract.isError(err[, "CODE"])`** â€” a type guard that narrows a caught error to the contract's errors (or a specific code), discriminated by `err.code`.

```ts
throw contract.error("CONFLICT", "duplicate transfer", { conflictingId: dup.id }); // server â†’ 409

if (contract.isError(err, "NOT_FOUND")) {
  /* err.code === "NOT_FOUND", err.status === 404 */
}
```

On the wire, every error is a fixed envelope â€” `{ kind: "HANDSHAKE", code, status, message, details }`. The `kind` brand lets the client tell a handshake error apart from any other non-OK response. `ApiError` mirrors that envelope exactly (`{ code, status, message, details }`, plus a client-only `response`), so the client reconstructs it 1:1. A non-handshake error response (a proxy, gateway, or non-handshake backend) is surfaced as an [`HttpError`](/handshake/client/fetch/#errors) instead â€” never an `ApiError`.

See [Error handling](/handshake/server/hono/#error-handling) for how the server surfaces them and [the client guide](/handshake/client/fetch/#errors) for catching them.

## Contract-level headers

Pass `headers` at the contract level to require headers across **every** route â€” it's merged into each route's own `headers` (route headers win on conflict):

```ts
const contract = createContract("/api", endpoints, {
  headers: Type.Object({ authorization: Type.String() }),
});
```

## Contract-level meta

Pass `meta` at the contract level to set a default metadata object across **every** route â€” it's shallow-merged into each route's own `meta` (route keys win on conflict):

```ts
const contract = createContract("/api", endpoints, {
  meta: { auth: true },
});
```

A route that declares its own `meta` overrides the matching keys and adds any extra ones; a route without `meta` inherits the default as-is.

## Composing Contracts into an Api

Handshake has a two-layer model. You **define** each resource's endpoints with `createContract` â€” typically one file per resource â€” and then **compose** those contracts into an api with `createApi`. A bare `Contract` is only a building block; the `Api` is the single unit the server adapter, fetch client, and OpenAPI generator consume.

`createApi` takes a **named record** of contracts (there is no array form) and returns the composed api:

```ts
// contracts/users.ts
export const usersContract = createContract("/users", {
  /* ... */
});

// contracts/posts.ts
export const postsContract = createContract("/posts", {
  /* ... */
});

// contracts/index.ts
import { createApi } from "@jayalfredprufrock/handshake/contract";
import { usersContract } from "./users";
import { postsContract } from "./posts";

export const api = createApi("/api", {
  users: usersContract,
  posts: postsContract,
});
```

The keys (`users`, `posts`) become **group names** â€” the server adapter implements one group at a time by name (e.g. `buildRoutes(api, "users", â€¦)`). Like `createContract`, the base path is the optional first argument (defaults to `"/"`); passing options requires an explicit base path. Each endpoint's `path` is prefixed with its source contract's `basePath`, so the final URL is `api.basePath + contract.basePath + endpointPath` (e.g. `/api` + `/users` + `/:id` â†’ `/api/users/:id`). Endpoint names must be unique across all groups â€” `createApi` throws on a duplicate.

Even a single resource is wrapped in an api â€” that api, not the contract, is what you hand to an adapter, the client, or the OpenAPI generator:

```ts
export const api = createApi("/api", { users: usersContract });
```

### Api-level errors, headers, and meta

`createApi` accepts the same `{ errors, headers, meta }` options as `createContract`, applied as cross-cutting defaults across **every** route in **every** group:

```ts
export const api = createApi(
  "/api",
  { users: usersContract, posts: postsContract },
  {
    errors: { UNAUTHORIZED: { status: 401 } },
    headers: Type.Object({ authorization: Type.String() }),
    meta: { auth: true },
  },
);
```

These merge with each contract's own errors/headers/meta, with the innermost value winning: route beats contract-default beats api-default. Declared errors from every group are merged too, and the api re-exposes `api.error(...)` / `api.isError(...)` typed over that full merged set.
