A contract is the single source of truth for your API. It defines every endpoint ŌĆö method, path, params, query, headers, body, and response ŌĆö plus the errors the API can return. Both the server adapter and the client are derived from it.

## Creating a Contract

`createContract(basePath?, endpoints, options?)`. The base path prefixes all endpoint paths (defaults to `"/"`).

```ts
import { createContract } from "@jayalfredprufrock/handshake/contract";
import { Type } from "typebox";

const contract = createContract("/api", {
  getUser: {
    method: "GET",
    path: "/users/:id",
    params: Type.Object({ id: Type.String() }),
    response: Type.Object({ id: Type.String(), name: Type.String() }),
  },
});
```

## Endpoint Definition

### `method`

**Required.** `"GET"`, `"POST"`, `"PATCH"`, or `"DELETE"`.

### `path`

**Required.** Path relative to the base path. Path parameters use `:param` syntax.

### `response`

**Required.** A TypeBox schema for the response body ŌĆö used for both type inference and runtime response validation.

### `responseCode`

**Optional.** The HTTP status for a successful response (defaults to `200`).

```ts
createUser: { method: "POST", path: "/users", body, response, responseCode: 201 }
```

### `params`

**Optional.** Schema for path parameters (a `Type.Object` keyed by `:param` segments). String values from the URL are coerced to match (e.g. `"42"` ŌåÆ `42`).

### `query`

**Optional.** Schema for query-string parameters. Coerced like params; array schemas support repeated keys (`?tags=a&tags=b`).

### `headers`

**Optional.** Schema for request headers. **Header names must be declared in lowercase.** Coerced like params; headers not named in the schema are ignored.

```ts
headers: Type.Object({ "x-api-key": Type.String() });
```

### `body`

**Optional.** Schema for the JSON request body. Validated strictly on the server ŌĆö missing or extra properties produce a `VALIDATION_ERROR` (400).

### `description` / `meta`

**Optional.** A human-readable description and an arbitrary metadata object for documentation or custom tooling.

## Errors

Errors are declared once on the contract as a **status-keyed map**. Each error body is discriminated by a `code` literal, and errors are **contract-wide** (any endpoint may return any of them):

```ts
const contract = createContract(
  "/api",
  { transfer: { method: "POST", path: "/transfer", body, response } },
  {
    errors: {
      401: Type.Object({ code: Type.Literal("UNAUTHORIZED") }),
      404: Type.Object({ code: Type.Literal("NOT_FOUND") }),
      409: Type.Object({ code: Type.Literal("CONFLICT"), conflictingId: Type.String() }),
    },
  },
);
```

Codes must be unique across statuses. The codes `VALIDATION_ERROR` and `UNKNOWN_ERROR` are **reserved** by the framework and throw if declared.

The contract exposes two typed helpers:

- **`contract.error("CODE", extra?)`** ŌĆö builds an `ApiError` with the status inferred from the code; `throw` the result. Undeclared codes and missing/extra fields are compile errors.
- **`contract.isError(err[, "CODE"])`** ŌĆö a type guard that narrows a caught error to the contract's error union (or a specific code).

```ts
throw contract.error("CONFLICT", { conflictingId: dup.id }); // server ŌåÆ 409

if (contract.isError(err, "NOT_FOUND")) {
  /* err.body: { code: "NOT_FOUND" } */
}
```

See [Error handling](/handshake/server/hono/#error-handling) for how the server surfaces them and [the client guide](/handshake/client/fetch/#errors) for catching them.

## Contract-level headers

Pass `headers` at the contract level to require headers across **every** route ŌĆö it's merged into each route's own `headers` (route headers win on conflict):

```ts
const contract = createContract(endpoints, {
  headers: Type.Object({ authorization: Type.String() }),
});
```

## Splitting Contracts Across Files

Define each resource's contract in its own file and combine them with `combineContracts`:

```ts
// contracts/index.ts
import { combineContracts } from "@jayalfredprufrock/handshake/contract";
import { usersContract } from "./users";
import { postsContract } from "./posts";

// array form (no named groups)
export const contract = combineContracts([usersContract, postsContract], { basePath: "/api" });

// named-object form ŌĆö keys become group names used by implementContract
export const named = combineContracts(
  { users: usersContract, posts: postsContract },
  { basePath: "/api" },
);
```

Each endpoint's `path` is prefixed with its source contract's `basePath`, so the final URL is `topBasePath + subBasePath + endpointPath`. Duplicate endpoint names throw. The options object also accepts contract-wide `errors` and `headers`, which merge across all combined contracts.
