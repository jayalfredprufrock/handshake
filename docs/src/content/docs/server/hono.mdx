The Hono adapter binds a contract to typed handlers and assembles a Hono app. It parses and validates requests, serializes responses, and gives you a uniform error envelope.

## Installation

```sh
npm install @jayalfredprufrock/handshake hono typebox
```

## Basic Usage

`implementContract` binds handlers to a contract and returns a `RouteModule`; `createHonoApp` assembles one or more modules into a `Hono` instance.

```ts
import { serve } from "@hono/node-server";
import { implementContract, createHonoApp } from "@jayalfredprufrock/handshake/hono";
import { contract } from "./contract";

const module = implementContract(contract, {
  getUser: ({ params }) => {
    const user = users.get(params.id);
    if (!user) throw contract.error("NOT_FOUND", "user not found");
    return user;
  },
  listUsers: () => [...users.values()],
});

const app = createHonoApp([module]);
serve({ fetch: app.fetch, port: 3000 });
```

`implementContract` throws immediately if any endpoint is missing a handler.

## `implementContract(contract, handlers | closure, options?)`

Pass a handler map, or a closure for per-group middleware:

```ts
const module = implementContract(contract, (group) => {
  group.use(bearerAuth({ token: process.env.API_TOKEN! }));
  group.implement("getUser", ({ params }) => ({ id: params.id, name: "Alice" }), {
    validateResponse: false, // per-handler override
  });
});
```

For a [combined contract](/handshake/core/contracts/#splitting-contracts-across-files) built with the named-object form, pass the group name first:

```ts
const usersModule = implementContract(contract, "users", {
  getUser: ({ params }) => ({ id: params.id, name: "Alice" }),
});
```

### Options

```ts
interface ImplementContractOptions {
  validateResponse?: boolean; // default true; governs both success and error response validation
  middleware?: EndpointMiddlewareFactory;
}
```

### Handler input

The input object contains only the fields the endpoint defines, plus `c`:

| Field     | Present when               | Type                   |
| --------- | -------------------------- | ---------------------- |
| `params`  | endpoint defines `params`  | `Static<E["params"]>`  |
| `query`   | endpoint defines `query`   | `Static<E["query"]>`   |
| `headers` | endpoint defines `headers` | `Static<E["headers"]>` |
| `body`    | endpoint defines `body`    | `Static<E["body"]>`    |
| `c`       | always                     | Hono `Context`         |

```ts
getUser: ({ params, headers, c }) => {
  const reqId = c.req.header("x-request-id");
  return { id: params.id, apiKey: headers["x-api-key"] };
},
```

### Return values

- **A plain value** matching the response schema — serialized as JSON (status from `responseCode`, default `200`).
- **A `Response`** — passed through as-is, bypassing response validation.

### Validation

- **Params, query, headers** are coerced to the schema; failures produce `VALIDATION_ERROR` (400).
- **Body** is validated strictly (no missing/extra properties).
- **Responses** are validated when `validateResponse` is on (the default) — unknown properties are stripped, and a mismatch is a server bug (see below).

## Error handling

Every error the server emits is an `ApiError` (`{ code, status, message, details }`). Throw a declared error with `contract.error("CODE", message, details?)` (or `new ApiError({ code, status, message, details })`); the `details` of a declared code are validated against its `details` schema before being sent. On the wire the response is a fixed envelope — `{ kind: "HANDSHAKE", code, status, message, details }` — matching the `ApiError` itself. The client reconstructs it, so `err.message` round-trips; the same `message` is available to `onError` as `err.message`.

A thrown `ApiError` whose `code` is **recognized** — declared in a contract, or a built-in `VALIDATION_ERROR`/`UNKNOWN_ERROR` — is serialized to its envelope **wherever it's thrown** (a handler, a middleware, a service), with no `onError` handling required. An `ApiError` with an _unrecognized_ code is treated as unknown and routed to `onError` (see below).

Two codes are **reserved** and emitted by the framework:

- **`VALIDATION_ERROR`** (400) — a request failed validation; `details` is a normalized `{ path?, keyword?, message }[]` array of issues.
- **`UNKNOWN_ERROR`** (500) — an unexpected error, or a handler response that failed validation. The cause is never sent to the client; it's logged on the server.

You can't _declare_ these in a contract's error map, but they're always throwable via `contract.error` (`contract.error("VALIDATION_ERROR", msg, issues?)` — the `issues` array is optional, e.g. `contract.error("VALIDATION_ERROR", "Phone number is invalid")` — and `contract.error("UNKNOWN_ERROR", msg)`) and matched by `contract.isError` — the client may receive them from any endpoint.

### `createHonoApp(modules, options?)`

```ts
interface CreateHonoAppOptions {
  onError?: (err: unknown, c: Context) => ApiError | void | Promise<ApiError | void>;
  middleware?: EndpointMiddlewareFactory | EndpointMiddlewareFactory[];
}
```

`onError` handles any error the framework didn't already serialize as a recognized contract error. A **known-code** `ApiError` is serialized automatically and never reaches `onError`; **everything else** does — plain exceptions, Hono `HTTPException`s, and `ApiError`s with an unrecognized code. Return an `ApiError` to shape the response, or return nothing for the default disposition:

- an `HTTPException` → keeps its own status (passed through),
- anything else → `UNKNOWN_ERROR` (500).

The server can **never** emit a non-`ApiError` body, regardless of what `onError` does (a throwing or malformed `onError` still yields a safe `UNKNOWN_ERROR`).

```ts
import { ApiError } from "@jayalfredprufrock/handshake/contract";
import { ResponseValidationError } from "@jayalfredprufrock/handshake/server";

const app = createHonoApp([module], {
  onError: (err) => {
    if (err instanceof ResponseValidationError) log.alert("contract drift", err.issues);
    if (err instanceof DatabaseError) metrics.inc("db_error");
    // return nothing → client gets { code: "UNKNOWN_ERROR" } (logged)
  },
});
```

Any error that isn't a recognized contract error reaches `onError` fully intact (custom fields, stack) while the client only ever sees the generic `UNKNOWN_ERROR` — so you can carry internal context on your own error classes and inspect it there. `ResponseValidationError` (from `/server`) is the framework's own example: it carries the validation `issues` for logging while the client sees `UNKNOWN_ERROR`.

### Framework and middleware errors

A Hono `HTTPException` — thrown by the framework or by middleware (e.g. bearer auth raising a `401`) — reaches `onError` like any other non-contract error. If `onError` doesn't return an `ApiError` for it, it **keeps its own status and response** (it is _not_ collapsed into `UNKNOWN_ERROR`), and the client surfaces it as a non-handshake [`HttpError`](/handshake/client/fetch/#errors) carrying that status. Because `onError` sees it first, you can convert it to a typed contract error if you prefer:

```ts
import { HTTPException } from "hono/http-exception";

const app = createHonoApp([module], {
  onError: (err) =>
    err instanceof HTTPException ? contract.error("UNAUTHORIZED", err.message) : undefined,
});
```

Only genuinely unexpected errors (bugs, a failed response validation) become `UNKNOWN_ERROR`.

## Global middleware

`createHonoApp` returns a plain `Hono` instance. For app-level middleware (logging, CORS), mount it on your own root Hono:

```ts
import { Hono } from "hono";
import { logger } from "hono/logger";

const root = new Hono();
root.use("*", logger());
root.route("/", createHonoApp([usersModule, postsModule]));
```

## Route ordering

Routes are sorted by path specificity before registration — literal segments take precedence over `:param` segments at the same position, so `/users/me` resolves to the literal handler regardless of definition order.
